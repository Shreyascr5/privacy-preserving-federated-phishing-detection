# SentinelFL — Privacy-Preserving Phishing Detection

SentinelFL is a demo-ready phishing investigation platform. A user supplies only a website URL and an IP address; the application automatically extracts security signals, queries a Flower-aggregated federated global model, and returns a risk score, confidence, verdict, and explanation.

> Every live prediction is generated by `models/federated_global_model.joblib`, the final Flower FedAvg aggregation. There is no manual engineered-feature form and no baseline-model fallback.

## What the system analyzes

The feature engine derives 19 normalized inputs: URL length, embedded IP, `@`, hyphens, suspicious keywords, subdomain depth, HTTPS, redirect patterns, suspicious TLD, entropy, complexity, URL shortening, punycode, digit density, IP validity, public/private/reserved scope, suspicious IP formatting, and domain/IP consistency. Optional live DNS checks are off by default so demonstrations remain fast and deterministic.

The risk score combines the federated model probability (72%) with transparent security rules (28%). The classifier verdict itself comes only from the federated global model.

## Federated learning implementation

- Four non-IID clients train local scikit-learn logistic classifiers.
- Raw URL/IP rows remain inside each client partition during local fitting.
- Client parameter deltas are L2-clipped and Gaussian noise is added before transmission.
- Selective updates skip non-improving client transmissions, with periodic synchronization.
- Flower `FedAvg.aggregate_fit` performs the global weighted aggregation.
- Participation, sent/skipped updates, training accuracy, loss, and communication reduction are saved as dashboard artifacts.

The included Gaussian mechanism is genuine update clipping/noising, but it intentionally makes **no formal epsilon guarantee** because no privacy accountant is included. This distinction is shown in the UI.

## Dataset

The old UCI Phishing Websites dataset contains 30 precomputed attributes but no raw URLs or IPs, so it cannot support honest direct URL inference. It is retained only as an archive under `dataset/processed/`.

`python -m src.build_url_dataset` creates a deterministic 4,000-record lexical URL/IP demonstration benchmark and four class-skewed client partitions. It is deliberately separable for a reliable classroom demo; its 100% held-out score is **not a production efficacy claim**.

For thesis-grade evaluation, use a maintained URL-level dataset such as PhiUSIIL, add timestamped IP mappings, and preserve a chronological holdout. The builder can ingest a replacement CSV without changing the application or federated pipeline:

```bash
python -m src.build_url_dataset \
  --source-csv /path/to/feed.csv \
  --url-column url \
  --ip-column ip_address \
  --label-column label
```

Labels must resolve to `0` for legitimate and `1` for phishing.

## Quick start (Python 3.13)

```bash
python3.13 -m venv venv
source venv/bin/activate
python -m pip install -r requirements.txt

python -m src.build_url_dataset
python -m federated.train_federated
python -m federated.evaluate_global

streamlit run streamlit_app/app.py
```

Open the displayed local address (normally `http://localhost:8501`). For a high-risk demonstration, use:

- URL: `https://paypal-login-security.xyz/verify/account`
- IP: `185.43.22.15`

For comparison:

- URL: `https://www.python.org/docs`
- IP: `8.8.8.8`

Run the automated checks with no additional test dependency:

```bash
python -m unittest discover -s tests -v
```

## Project structure

```text
phishing_detection/       feature extraction, model, risk, history, configuration
src/build_url_dataset.py  reproducible dataset creation / external CSV ingestion
federated/                Flower FedAvg training and global evaluation
streamlit_app/            six-page Streamlit dashboard
dataset/url_ip/           direct URL/IP benchmark and client partitions
models/                   deployable global model and FL metadata
results/                  metrics, plots, participation, and local history
tests/                    feature and end-to-end prediction checks
```

All runtime paths are derived with `pathlib`; commands may be launched from the repository root without machine-specific paths.

## Professor-ready explanation

1. The URL and IP are converted to the exact same normalized feature vector used during training.
2. Each client trains locally on a different class distribution; raw records are not aggregated.
3. Updates are clipped, noised, and selectively transmitted.
4. Flower FedAvg computes a sample-weighted global parameter set.
5. That serialized global model produces the displayed probability and verdict.
6. Coefficient contributions and independent security checks explain the result.

## Deployment

For Streamlit Community Cloud, set `streamlit_app/app.py` as the entry point and use Python 3.13. Commit the generated model and result artifacts required by the dashboard. In production, run training in a trusted pipeline, store the signed model artifact in versioned object storage, disable local history or encrypt it, and move DNS enrichment to a rate-limited service with strict timeouts.
